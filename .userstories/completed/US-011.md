## Implementation Summary

The `TemplatesController` already had stub letter upload endpoints from US-010, but they were not
correctly wiring the uploaded DOCX stream to the service layer. Both `CreateLetterTemplate` (POST)
and `UpdateLetterTemplate` (PUT) were updated to open the `IFormFile` stream and pass it as
`DocxContent` to `CreateTemplateRequest`/`UpdateTemplateRequest`, enabling the service to persist
the DOCX file and set the correct versioned `BodyPath`. Input validation was added for `name`
(required, max 200 chars) and `description` (max 500 chars). HTTP 409 on name collision is handled
at the controller level by catching `ValidationException` containing "already exists". HTTP 422 on
channel mismatch is enforced by pre-checking the existing template's channel before calling
`UpdateAsync`. The `RequireDesignerOrAdmin` policy is applied on both endpoints, excluding
`CampaignManager`. `TemplateDto.BodyPath` already returns only the relative storage path
(`templates/{id}/vN.docx`). 23 API integration tests were added covering create/update success
paths and all error cases (400, 404, 409, 413, 422). API documentation was added at
`docs/letter-template-upload-api.md`.
