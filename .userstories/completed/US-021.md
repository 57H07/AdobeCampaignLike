### Notes:
- `LetterDispatcher` is stateful within a campaign batch — it accumulates per-recipient PDFs via `SendAsync`, then consolidates and writes via `FlushBatchAsync`.
- PDF generation reuses existing `LetterPostProcessor` (DinkToPdf, US-013). Native `libwkhtmltox.dll` must be present in the application root on Windows Server.
- Consolidation uses existing `PdfConsolidationService` (PdfSharp, US-013). Max 500 pages per batch.
- `PrintProviderFileDropHandler.WriteAsync` is `virtual` to allow test overrides and future API-based provider implementations (resolving Open Question Q5).
- File naming convention: `CAMPAIGN_{campaignId:N}_{yyyyMMddHHmmss}_{batchNumber:D3}.pdf`
- CSV manifest: `SequenceInBatch,RecipientId,DisplayName,PageCount,BatchFileName` — RFC 4180 compliant.
- 28 unit tests covering: happy path, channel disabled, empty content, post-processor failures, consolidation, page order preservation, manifest generation and CSV escaping.
- Open Question Q5 (print provider format) addressed via `virtual WriteAsync` hook point.
